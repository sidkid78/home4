- # Orchestrator Execution Plan: HOMEase Platform

An analysis of the HOMEase repository and the blueprint in `specs/info.md` reveals that the project is in its inception stage. The repository currently contains empty structural folders (`commands/`, `workflows/`, `specs/current`, `specs/archive`) and a comprehensive architectural specification file (`specs/info.md`).

The blueprint defines a 6-step workflow (**Capture, Assess, Report, Prioritize, Match, Execute**) designed to transition HOMEase from a Lead Marketplace into a Longitudinal Property Intelligence Platform.

Here is the detailed architectural evaluation and step-by-step execution plan.

---

## 1. Summary of What We Found

1. **Repository State:** 
   - Clean slate with zero source files.
   - The directory structure is established for an **Orchestrator-Workers pattern**.
   - `specs/info.md` contains excellent architectural designs across 8 distinct specialized roles.
2. **Tech Stack & Integrations:**
   - **Frontend:** React, Vite, TypeScript, Tailwind CSS, XState, Framer Motion, Supabase client, local `IndexedDB` caching (`idb-keyval`).
   - **Backend:** Fastify (Node.js/TypeScript) with Ajv JSON Schema validation, Prisma ORM, Redis/BullMQ for async task queue.
   - **Database & Storage:** PostgreSQL with Supabase Auth, Row-Level Security (RLS) policies, and Supabase Storage (Private/Public buckets).
   - **AI Pipeline:** Google Gemini (multimodal image analysis) with Human-in-the-Loop (HITL) Expert Review.
   - **Payments & FinTech:** Stripe Connect (Express onboarding model) and Stripe Checkout.
   - **Enterprise compliance:** HIPAA-ready auditing, HL7 FHIR observation mapping, and the Lifetime Home Record (LHR) longitudinal dataset.

---

## 2. Key Requirements Identified

1. **High-Fidelity Space Capture (Mobile-First UI):** Must prevent "garbage-in" scan data from non-technical users through strict frontend state management (XState), camera resolution management, and direct-to-storage uploads.
2. **AI-Driven Risk Identification:** Multimodal image reasoning to identify fall risks, estimate dimensions, and generate structured JSON outputs. Must trigger a Human-in-the-Loop (HITL) review dashboard when confidence scores fall below 0.75.
3. **Safety & Economic Justification:** An ROI engine that contrasts the cost of modifications against the actuarial cost of a home fall ($50,000 baseline) and converts the findings into an automated Bill of Quantities (BoQ) material checklist.
4. **Stripe Concurrency & Security Paywalls:** 
   - Absolute concurrency controls (atomic DB transaction with locking) to prevent two contractors from purchasing the same exclusive lead.
   - Row-Level Security (RLS) policies that hide Property PII and report details until a $125 Lead purchase is settled.
5. **Lifetime Home Record (LHR) & FHIR Mapping:** Interoperable data structure mapping safety improvements into HL7 FHIR Observation standards, laying a foundation for future healthcare integration.

---

## 3. Recommended Approach

### A. Critical Technical Correction: Upgrade to `@google/genai`
The original spec in `specs/info.md` references the legacy SDK `@google/generative-ai` and the deprecated model `gemini-1.5-pro`. 
* **Correction:** We will strictly implement the new **Google GenAI SDK** (`@google/genai` for Node.js) using the current standard models: **`gemini-3.5-flash`** for general multimodal analysis and **`gemini-3.1-pro-preview`** for complex spatial/structural reasoning.
* **JS Code Pattern upgrade:**
  ```typescript
  import { GoogleGenAI } from '@google/genai';
  const ai = new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY });
  const response = await ai.models.generateContent({
    model: 'gemini-3.5-flash',
    contents: [imagePart, textPrompt],
    config: {
      responseMimeType: 'application/json',
      responseSchema: myPydanticOrJsonSchema,
      systemInstruction: 'You are a Certified Aging-in-Place Specialist (CAPS)...'
    }
  });
  ```

### B. Decoupled Service-Oriented Development
To prevent monolithic bottlenecks, the development will follow a clean database-first sequence:
1. **Data Layer:** Schema and RLS definitions.
2. **API & Orchestration Layer:** Fastify endpoints and asynchronous processing (BullMQ queue).
3. **AI Layer:** Custom prompt engineering, scaling rules, and structured schema validations.
4. **Frontend Interface:** XState capture workflow, media handlers, direct upload, and a high-contrast demographic-friendly design.
5. **FinTech Layer:** Concurrency control, Stripe connect onboarding, webhooks, and ledger entries.
6. **Enterprise Interoperability:** Analytical health scoring and FHIR integrations.

## 4. Detailed Step-by-Step Execution Plan

### Phase 1: Database & Security Foundations (Data Layer)
* **Goal:** Set up Postgres database with Prisma and configure security policies.
* **Steps:**
  1. Initialize `package.json` in root and backend workspace. Configure database credentials in `.env`.
  2. Setup Prisma Schema `prisma/schema.prisma` with defined schemas for `User`, `Property`, `Capture`, `Assessment`, `Report`, `Lead`, `Transaction`, `PropertyHealthScore`, `ModificationLedger`, and `EnterprisePartner`.
  3. Generate migrations and apply them.
  4. Write a PostgreSQL migration script to enable Row-Level Security (RLS) and apply policy rules for:
     - Homeowner access (access only owned property and captures).
     - Contractor access (access available leads, access full report only if lead status is `SOLD` and contractor ID matches).
     - Admin access (bypass rules for QA validation).

### Phase 2: Backend Core API & State Machine (Fastify Service)
* **Goal:** Create a high-performance HTTP gateway and an asynchronous job manager.
* **Steps:**
  1. Initialize Fastify server with Typebox or Ajv schema compilation.
  2. Implement `POST /v1/captures` with strict JSON validation matching the room constraints.
  3. Configure a BullMQ queue processor with Redis backing. When a capture is saved with status `PENDING`, enqueue an asynchronous `assessment_job`.
  4. Implement `GET /v1/reports/:id` with integrated authorization checks that query RLS permissions or enforce contractor lead-sold requirements.
  5. Set up global error handling, 30-second API timeouts, and request idempotency logic.

### Phase 3: Modernized Gemini AI Pipeline (AI/ML Service)
* **Goal:** Build the CAPS-persona multimodal intelligence engine using the correct Google GenAI SDK.
* **Steps:**
  1. Install `@google/genai` in backend dependencies.
  2. Develop `src/services/ai_engine.service.ts` to fetch images from Supabase Storage and format them for `gemini-3.5-flash` or `gemini-3.1-pro-preview`.
  3. Engineer the **Certified Aging-in-Place Specialist (CAPS)** prompt with explicit:
     - **Chain-of-Thought (CoT)** reasoning steps.
     - **Reference Scaling rules:** Use standard tiles (4-inch), outlet heights, or toilet paper fixtures to calculate door widths and counter heights.
     - **Structured output schema** without `additionalProperties` (Gemini strict format).
  4. Implement the **HITL Trigger Logic**: Route records to an `ExpertReviewQueue` table if the model confidence is `< 0.75` or critical structural dimensions flag high variance.

### Phase 4: Guided Mobile Capture Interface (Client Experience)
* **Goal:** React mobile-web guided-capture SPA.
* **Steps:**
  1. Scaffolding for React-Vite-TypeScript with Tailwind CSS.
  2. Implement the `useCamera` hook leveraging raw `navigator.mediaDevices.getUserMedia` for capturing high-res (1080p ideal) JPEG blobs.
  3. Define `captureMachine` in XState to handle permission checks, camera alignment, real-time visual instructions via SVG overlay, offline caching to `IndexedDB`, and retry routines.
  4. Build the direct-to-Supabase Storage upload logic. The frontend uploads media to Supabase buckets first, then posts metadata to Fastify `POST /v1/captures`.
  5. Implement demographic UI optimizations (minimum `text-lg` size, high contrast ratio, large touch targets, and haptic warnings).

### Phase 5: Reporting & Prioritization Engine (Analytics Layer)
* **Goal:** Calculate weighted scores, ROI value, and Bill of Quantities (BoQ) material checklists.
* **Steps:**
  1. Write the **Weighted Safety Score** function:
     $$PriorityScore = (Severity \times 0.6 + RoomImpact \times 0.4) \times MobilityMultiplier$$
  2. Implement the **ROI Calculator**: Determine total project cost vs. the avoided $50,000 cost of a fall (weighted by the risk reduction factor).
  3. Create the `MaterialMapper` lookup system to expand AI recommendations (e.g., "Raise toilet height") into complete checklists (Comfort-height toilet, wax ring, floor bolts).
  4. Implement the report rendering service (saving structured data in PostgreSQL JSONB and optionally compiling a PDF package).

### Phase 6: FinTech Marketplace & Atomic Lead Transactions (Stripe Layer)
* **Goal:** Secure onboarding, split payments, and double-purchase concurrency guards.
* **Steps:**
  1. Integrate Stripe SDK and configure Connect Express links (`/v1/contractors/onboard`).
  2. Implement the checkout API `/v1/leads/:id/checkout` wrapped in a strict **database-level transaction (`prisma.$transaction`)** using a **`SELECT FOR UPDATE`** lock to prevent concurrency race-conditions (lead sniping).
  3. Implement Stripe webhook server in Fastify to listen for `checkout.session.completed` events and update lead statuses atomically.
  4. Set up the transaction ledger to record gross amount, platform take, and net payouts.

### Phase 7: Longitudinal Records & Compliance Ready (Enterprise Layer)
* **Goal:** Complete compliance pipelines, LHR score trend updates, and FHIR outputs.
* **Steps:**
  1. Write scoring trend logic in `PropertyHealthScore` table that records safety improvements when a contractor completes an modification.
  2. Create the HL7 FHIR Mapping utility mapping safety scores, risks, and PDFs to standard `Observation`, `Condition`, and `DocumentReference` FHIR resources.
  3. Implement comprehensive `AccessAudit` ledger records tracking who accessed what PII for SOC 2 Type II compliance.

## 5. Potential Challenges and Solutions

| Challenge | Impact | Mitigation / Solution |
| :--- | :--- | :--- |
| **User Camera Drift / Low Light** | Blurry imagery, failing AI spatial calculations. | **XState Capture Quality Cues:** Accelometer velocity sensing via JavaScript alerts the user "Panning too fast." Display warnings on low light / low resolution prior to submission. |
| **AI Measurement Hallucinations** | Faulty door width estimates could lead to bad contractor materials. | **Anchor Scaling + HITL Safeguard:** Prompt Gemini to explicitly output its scaling reference anchor. If confidence interval is "low", flag the lead for mandatory HITL manual override. |
| **Lead Concurrency (Lead Sniping)** | Two contractors charging credit cards for the same exclusive lead simultaneously. | **Pessimistic DB Locks:** Ensure database row lock (`FOR UPDATE`) is acquired before Stripe transaction session is created. Reject double checkouts immediately. |
| **HIPAA / PHI Exposure** | Severe compliance penalty. | **Clean Property Isolation:** Separate user profile fields from property records. Use Lead proxies for marketplace matching. Encrypt sensitive fields with pg_crypto. |
